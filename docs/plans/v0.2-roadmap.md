## Remaining Work Units

### Phase A: Reliability & Polish (Low risk, high value)

#### A1 — Fix TUI raw mode crash
**Problem:** `startTui()` calls Ink's `render()` which requires `process.stdin.setRawMode()`. This throws in non-interactive shells (CI, piped processes, background daemon mode).
**Fix:** Guard with `process.stdin.isTTY && process.stdin.setRawMode` check before rendering. Fall back to plain terminal output if not available. Ink supports `stdin: false` option in render config — investigate using that.
**Files:** `src/tui.tsx`, `src/cli.ts` (the `--tui` branch)
**Effort:** Small

#### A2 — Update deprecated model default
**Problem:** `models.routing` defaults to `claude-3-5-haiku-20241022` which is deprecated (EOL Feb 2026). CommsAgent uses this for summarization.
**Fix:** Update default in `src/config.ts` and `src/modelSelector.ts` to `claude-haiku-4-5-20251001`.
**Files:** `src/config.ts` (line 76), `src/modelSelector.ts` (line 16)
**Effort:** Trivial

#### A3 — Harden commsAgent for offline/keyless use
**Problem:** CommsAgent.summarize() tries to call the Anthropic API. In environments without `ANTHROPIC_API_KEY`, it falls back to raw formatting — but logs a console.warn about the deprecated model first, and the API call adds latency.
**Fix:** Check for `ANTHROPIC_API_KEY` before attempting the API call. Skip straight to rawFormat if no key is set.
**Files:** `src/commsAgent.ts` (summarize method, ~line 117)
**Effort:** Small

#### A4 — Clean up CONTRIBUTING.md
**Problem:** Still references co-author trailers. User explicitly doesn't want those.
**Fix:** Remove the co-author section from CONTRIBUTING.md.
**Files:** `CONTRIBUTING.md`
**Effort:** Trivial

#### A5 — Update AGENTS.md
**Problem:** Architecture diagram and key files table are stale — missing 15+ modules added since initial scaffold.
**Fix:** Mirror the architecture section from README.md. Add pipeline, governance, commsAgent, memory, sanitizer, etc. to key files table.
**Files:** `AGENTS.md`
**Effort:** Small

### Phase C: Features (New capabilities)

#### C1 — Web dashboard (SSE stream)
**What:** The agentOutput emitter already has `setSseBroadcast()` wired up but nothing serves SSE. Add an SSE endpoint at `GET /events` that streams DriveOutputEvents to a browser.
**Why:** The TUI is great for interactive terminals, but a web dashboard would let you watch operators from any browser tab, phone, etc.
**Approach:**
1. Add `GET /events` to the HTTP server in `mcpServer.ts` that writes SSE headers and pipes agentOutput events
2. Ship a static `public/index.html` with EventSource that renders activity, operators, and mode
3. Config key: `agentScreen.webPort` (already defaulted to 7892) or reuse main port
**Effort:** Medium
**Files:** `src/mcpServer.ts`, new `public/index.html`

#### C2 — Operator task chaining / DAG execution
**What:** Currently operators are independent. Add a `drive_run_dag` MCP tool that accepts a dependency graph of tasks and dispatches them in order.
**Why:** The plan you just executed (Units 1-9 with dependencies) was manually orchestrated. claude-drive should be able to run that kind of workflow natively.
**Approach:**
1. New `src/dagRunner.ts` — accepts `{ tasks: [{ id, task, role?, dependsOn: string[] }] }`, topological sort, spawn operators, await completion
2. New MCP tool `drive_run_dag` in `mcpServer.ts`
3. Track DAG execution state in the registry (new `dagId` field on operators)
**Effort:** Large
**Dependencies:** Needs reliable operator completion detection (currently runOperator is fire-and-forget via `void`)

#### C3 — Operator completion tracking
**What:** `runOperator()` returns void (fire-and-forget). There's no way to know when an operator finishes.
**Why:** Blocks DAG execution (C2), comms agent can't report completion accurately, no way to auto-dismiss finished operators.
**Approach:**
1. Track a Promise per operator in the registry (`completionPromise`)
2. `runOperator()` resolves it on finish (success or error)
3. Add `operator_await` MCP tool that blocks until operator completes
4. Auto-update operator status to "completed" on finish
**Effort:** Medium
**Files:** `src/operatorManager.ts`, `src/operatorRegistry.ts`, `src/mcpServer.ts`

#### C4 — Voice input (STT → pipeline)
**What:** The system has TTS (output) but no STT (input). Add Whisper-based voice input.
**Why:** The project is called "voice-first" but only has the output half.
**Approach:**
1. New `src/stt.ts` — Whisper.cpp or OpenAI Whisper API for transcription
2. `src/voiceLoop.ts` — VAD (voice activity detection) → record → transcribe → pipeline → TTS response
3. New CLI flag: `claude-drive start --voice`
4. Config: `voice.enabled`, `voice.whisperPath` (already in defaults but unused)
**Effort:** Large
**Dependencies:** Whisper binary or API key

#### C5 — Operator memory persistence across sessions
**What:** When you restore a session, operator memory (`op.memory[]`) comes back, but it's just string arrays. No semantic structure.
**Why:** Long-running operator teams should accumulate and share knowledge.
**Approach:**
1. Extend `sessionMemory.ts` to namespace entries by operatorId
2. On session restore, re-inject each operator's memory context into their system prompt
3. New MCP tool `operator_memory_search` for cross-operator memory queries
**Effort:** Medium

#### C6 — npm publish + global install
**What:** The publish workflow exists but there's no npm package published yet.
**Why:** `npm i -g claude-drive` would be the easiest install path.
**Approach:**
1. Set up NPM_TOKEN secret on GitHub
2. Tag a release: `git tag release/v0.1.0 && git push --tags`
3. Add `"files"` field to package.json to whitelist published files
4. Test with `npm pack` first
**Effort:** Small

### Phase D: Infrastructure & Quality

#### D1 — CI: add integration test to workflow
**Problem:** `ci.yml` runs `npm test` which includes integration tests, but those start a real HTTP server. Verify this works reliably in GitHub Actions.
**Action:** Run CI and check if the integration test passes. If flaky, add retry or increase jest timeout.

#### D2 — E2E test: start → tool calls → stop
**What:** Automate what we just did manually in the demo.
**Approach:** New `tests/integration/e2e-lifecycle.test.ts` that:
1. Spawns `node out/cli.js start` as a child process
2. Waits for health check
3. Runs MCP tool calls (spawn, list, pipeline_process, etc.)
4. Calls `POST /shutdown`
5. Verifies process exits

#### D3 — Add `--daemon` flag (background mode)
**What:** `claude-drive start` keeps the foreground. Users need a way to background it.
**Approach:** Add `--daemon` flag that forks the process via `child_process.fork()` and detaches.
**Files:** `src/cli.ts`

#### D4 — Log file rotation
**What:** PersistentMemory daily logs accumulate forever. `pruneOlderThan()` exists but is never called automatically.
**Approach:** Call `pruneOlderThan(config.persistentMemory.retentionDays)` on startup.
**Files:** `src/cli.ts` (in the `start` action), or `src/mcpServer.ts`

#### D5 — Graceful shutdown: drain active operators
**What:** `POST /shutdown` exits immediately. Active operators get killed mid-task.
**Approach:** On shutdown, set a "draining" flag, wait for active operators to finish (with timeout), then exit.
**Files:** `src/mcpServer.ts` (shutdown handler)

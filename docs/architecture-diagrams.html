1. High-Level System Overview

The 30,000-foot view. claude-drive is a daemon exposing an MCP server. Claude Code connects to it as a tool provider. Operators (AI subagents) are orchestrated through a registry, each with its own AbortController, memory context, and optional git worktree.

3. Operator Lifecycle State Machine

Operators are named AI subagents. Only ONE can be Active (foreground) at a time. Dismiss fires AbortController and cascades to children.

stateDiagram-v2
    [*] --> Spawned: operator_spawn

    Spawned --> Active: switchTo()
    Spawned --> Background: (not foreground)

    Active --> Background: switchTo(other)
    Background --> Active: switchTo(this)

    Active --> Paused: pause()
    Background --> Paused: pause()
    Paused --> Active: resume()
    Paused --> Background: resume()

    Active --> Completed: dismiss()
    Background --> Completed: dismiss()
    Paused --> Completed: dismiss()

    Active --> Merged: worktree_merge
    Background --> Merged: worktree_merge

    Completed --> [*]
    Merged --> [*]

4. Task Dispatch Flow

When Claude Code calls drive_run_task, the full flow through maxConcurrent checks, AbortController setup, memory context injection, and SDK query loop.

flowchart TD

    B -->|No| C["Find or spawn operator"]

    C --> D["runOperator(op, task, opts)"]

    D --> G["buildOperatorSystemPrompt()\nrole + memory + permissions"]

    E --> H
    F --> H
    G --> H["SDK query(prompt, tools, system, agents)"]

    H --> I{"for await msg of query"}

    I -->|"result"| J["Extract costs:\ntotal_cost_usd, duration_ms, num_turns"]
    I -->|"rate_limit"| K["Log rate limit event"]
    I -->|loop| L{"signal.aborted?"}

    L -->|Yes| M["Log 'Task cancelled'\nbreak loop"]
    L -->|No| I

    J --> N["Fire TaskComplete hook"]
    N --> O["registry.recordTaskStats()"]
    O --> P["flushStatus() -> status.json"]

5. Memory System & Auto-Dream

Typed memory with confidence decay. Auto-dream runs every 15 minutes to prune, decay, merge similar entries, and promote operator-local knowledge to global.

flowchart LR
    subgraph Write ["Write Path"]
        R["remember(opId, kind, content, tags)"] --> MS["memoryStore\n~/.claude-drive/memory.json"]
        C["correct(opId, oldId, newContent)"] --> MS
        S["shareMemory(id)"] -->|"remove operatorId\n(make global)"| MS
    end

    subgraph Read ["Read Path"]
        RC["recall(opId, query)"] --> Q["query(kinds, tags, search)"]
        Q --> MS
        BC["buildMemoryContext(opId)"] --> MS
        BC -->|"Top 15 entries"| SP["System Prompt Injection"]
    end

    subgraph Dream ["Auto-Dream (every 15min)"]
        AD["AutoDreamDaemon"] --> P1["1. Prune expired +\nconfidence below 0.2"]
        P1 --> P2["2. Decay: exponential\nhalf-life 168h"]
        P2 --> P3["3. Merge: 70%+ keyword\noverlap -> supersede"]
        P3 --> P4["4. Promote: operator-scoped\nseen in 2+ operators -> global"]
    end

    AD --> MS

6. Safety & Approval Gates

Pattern-based command filtering. Block/warn/log patterns checked before execution. Per-operator throttling prevents runaway operators.

flowchart TD
    A["Tool use (e.g., Bash command)"] --> B["getGateResult(cmdText, opId)"]

    B --> C{"Match block patterns?\nrm -rf, format c:, del /f"}
    C -->|Yes| D["action: BLOCK"]

    C -->|No| E{"Match warn patterns?\nforce push, hard reset, drop db"}
    E -->|Yes| F["action: WARN"]

    E -->|No| G{"Match log patterns?\nsudo, npm publish, git push"}
    G -->|Yes| H["action: LOG"]
    G -->|No| I["action: ALLOW"]

    D --> J["approvalQueue.requestApproval()"]
    F --> J

    J --> K{"User responds\nwithin 30s?"}
    K -->|Approved| L["Execute tool"]
    K -->|Denied| M["Abort tool"]
    K -->|Timeout| M

    H --> L
    I --> L

    D --> T["recordAction(opId, 'block')"]
    T --> U{">= 3 blocks OR >= 5 warns?"}
    U -->|Yes| V["Operator THROTTLED\nAll actions blocked"]

7. Persistence Architecture

All writers go through atomicWriteJSON() — write to .tmp then rename. Atomic on POSIX and NTFS. No partial writes, no corruption on crash.

    Ct -->|command| CMD["Execute shell command\nContext passed as env vars"]

    CMD --> X{"Exit code?"}
    X -->|0| OK["Continue"]

10. Worktree Isolation

Each operator gets its own git branch and working directory. Promise-chain mutex prevents race conditions.

sequenceDiagram
    participant MCP as MCP Tool
    participant WM as WorktreeManager
    participant Git as Git
    participant REG as OperatorRegistry

    MCP->>WM: allocate(opId, baseRef?)
    WM->>WM: Acquire mutex lock
    WM->>Git: git branch drive/op/opId [baseRef]
    WM->>Git: git worktree add .drive/worktrees/opId/ drive/op/opId
    WM-->>MCP: { worktreePath, branchName }
    MCP->>REG: updateWorkspaceState(opId, { path, branch })

    Note over WM,Git: Operator works in isolated worktree...

    MCP->>WM: release(opId)
    WM->>WM: Acquire mutex lock
    WM->>Git: git worktree remove .drive/worktrees/opId/
    WM->>Git: git branch -D drive/op/opId
